How hashing works
A cryptographic hash is a one-way fingerprint. The same input always produces the same fixed-length output, but you cannot reverse it back to the original. Flip a single character and the entire digest changes — the avalanche effect. Hashes verify file integrity, store passwords (with salt), and power digital signatures. SHA-256 is today's baseline; SHA-1 and MD5 are shown for legacy comparison only — never use them for security.
Encoding is not encryption
Encoding reshapes data so it can travel safely through text-only systems (emails, URLs, JSON). It is fully reversible by anyone and offers zero secrecy. Base64 maps 3 bytes to 4 ASCII characters; Hex shows each byte as two digits; URL escapes unsafe characters. If you need actual confidentiality, use the AES Vault tab instead.
Real AES-256-GCM, in your browser
This is genuine authenticated encryption. Your passphrase is stretched into a 256-bit key with PBKDF2 (250,000 SHA-256 rounds + a random salt), then AES-GCM encrypts the message with a fresh random IV. The salt and IV are packed with the ciphertext so decryption is self-contained. GCM also authenticates — tamper with one byte and decryption fails loudly. Lose the passphrase and the data is gone; there is no backdoor.
Classic ciphers — great to learn, unsafe to trust
Caesar shifts each letter by a fixed amount; ROT13 is Caesar-13; Atbash mirrors the alphabet; Vigenère uses a repeating keyword for a shifting shift; XOR combines each byte with a key byte. These are the historical roots of cryptography and perfect for teaching substitution and modular arithmetic — but every one of them falls to frequency analysis or a short brute force. Use them to understand, then reach for AES for anything real.

What actually makes a password strong
Strength is measured in entropy — the number of guesses an attacker must try. Entropy grows with both length and the size of the character pool, so a long passphrase beats a short jumble of symbols. Our generator uses crypto.getRandomValues(), a cryptographically secure random source — not Math.random(). The crack-time estimate assumes a fast offline attacker at ~10 billion guesses/second.
RSA — the maths behind public-key crypto
RSA uses a key pair: a public key you share and a private key you guard. Anyone can encrypt to you with your public key, but only your private key can decrypt (RSA-OAEP). Flip it around and you can sign with your private key so anyone can verify authenticity with your public key (RSA-PSS) — try editing a signed message and re-verifying to watch it fail. Keys are 2048-bit, generated locally, and exist only for this session. Note: RSA encrypts small payloads only; real systems use it to wrap an AES key (hybrid encryption). Learn more on the Learn page.
Privacy by design (k-anonymity): your browser SHA-1 hashes the password and sends only the first 5 characters of that hash to Have I Been Pwned. The full password — and the rest of the hash — never leave this tab.
How the breach check works
Have I Been Pwned holds a database of over 800 million passwords exposed in real data breaches. Using k-anonymity, your browser hashes the password with SHA-1, sends just the 5-character hash prefix, and receives back every matching suffix and its breach count — then checks locally whether yours is among them. The service never sees your password or even its full hash. If a password appears here, it's public knowledge to attackers — stop using it everywhere and generate a fresh one in the Passwords tab. This is the one tool that reaches the network; every other tool is fully offline.
