01 Hashing

A cryptographic hash function takes any input — a word, a file, a movie — and produces a fixed-length string called a digest. The same input always yields the same digest, but the function is one-way: you cannot run it backwards to recover the input.

The magic property is the avalanche effect: change a single bit of the input and roughly half the output bits flip. That's what makes hashes useful for integrity — download a file, hash it, and compare against the published digest; if even one byte was altered in transit, the digests won't match.

Where hashes are used

  • Password storage — servers store a salted hash, never the password itself.
  • File & message integrity — checksums and content addressing (e.g. Git commits).
  • Digital signatures — you sign the hash of a document, not the whole document.
Not all hashes are equal. MD5 and SHA-1 are broken — attackers can craft collisions. Use SHA-256 or better. For passwords specifically, use a slow hash like bcrypt, scrypt or Argon2, not a raw SHA.

02 Encoding

Encoding is not encryption. This is the single most common misconception in security. Encoding reshapes data into a format that survives text-only channels — email bodies, URLs, JSON — but anyone can reverse it. There is no key and no secrecy.

  • Base64 — maps every 3 bytes to 4 printable ASCII characters. Great for embedding binary in text.
  • Hex — writes each byte as two hexadecimal digits; common in fingerprints and debugging.
  • URL encoding — escapes characters that have special meaning in a URL (spaces, &, ?).
Rule of thumb: if you can decode it without a secret, it was never protecting anything. Need confidentiality? Reach for AES.

03 Symmetric encryption — AES

Symmetric means the same key encrypts and decrypts. AES (Advanced Encryption Standard) is the world's workhorse cipher — it protects TLS traffic, disk encryption and messaging apps. CipherVault uses AES-256-GCM, a 256-bit key in Galois/Counter Mode.

The two ingredients that make it safe

  • Key derivation (PBKDF2). Humans pick weak passphrases, so we stretch yours through 250,000 rounds of hashing with a random salt. This makes each guess expensive for an attacker and ensures the same passphrase produces a different key each time.
  • Authenticated encryption (GCM). Beyond hiding the message, GCM attaches an authentication tag. If a single byte of ciphertext is altered, decryption fails loudly instead of returning garbage — you get confidentiality and tamper-detection.

A fresh random IV (initialization vector) is used every time, so encrypting the same message twice yields different ciphertext. CipherVault packs the salt, IV and ciphertext into one Base64 blob so decryption is fully self-contained.

There is no backdoor. Lose the passphrase and the data is unrecoverable — that's the point of strong encryption.

04 Public-key cryptography — RSA

Symmetric encryption has one hard problem: how do two people agree on a shared key without meeting? Public-key cryptography solves it with a key pair — two mathematically linked keys where what one locks, only the other unlocks.

  • Encryption (RSA-OAEP). You publish your public key. Anyone can encrypt a message to you with it, but only your private key can decrypt. No shared secret required.
  • Signatures (RSA-PSS). Reverse the roles: you sign a message with your private key, and anyone can verify it with your public key. This proves authenticity and integrity — edit the message after signing and verification fails.

RSA is slow and can only encrypt small payloads (a few hundred bytes), so real systems use hybrid encryption: RSA encrypts a random AES key, and AES encrypts the actual data. You get the key-exchange convenience of RSA with the speed of AES.

Try the tamper test: in the toolkit, sign a message, then change one character and hit Verify. Watch it flip from valid to invalid — that's a digital signature protecting integrity.

05 Classic ciphers

Before computers, cryptography was pen-and-paper letter substitution. These ciphers are wonderful teaching tools because you can trace every step by hand — and instructive failures, because every one of them is trivially broken today.

  • Caesar — shift each letter a fixed number of places. Only 25 possible keys, so a brute force is instant.
  • ROT13 — Caesar with a shift of 13; applying it twice returns the original. Used for hiding spoilers, not secrets.
  • Atbash — mirror the alphabet (A↔Z, B↔Y). A fixed substitution with no key at all.
  • Vigenère — a repeating keyword sets a different shift for each letter. Unbroken for centuries, then defeated by spotting the repeating key length.
  • XOR — combine each byte with a key byte. Secure only as a one-time pad (random key as long as the message, never reused); with a short repeating key it's easily cracked.
Never trust a classic cipher with real data. They fall to frequency analysis or brute force in seconds. Learn from them, then use AES.

06 Passwords & entropy

Password strength isn't about weird symbols — it's about entropy, the number of guesses an attacker must try, measured in bits. Each extra bit doubles the work. Entropy grows with both the length and the size of the character pool, which is why a long passphrase beats a short cryptic jumble.

Roughly: entropy ≈ length × log₂(pool size). A 20-character password drawn from all keyboard characters lands well over 100 bits — effectively uncrackable by brute force. A clever 8-character password does not.

  • Use a generator. Truly random beats human-chosen every time. CipherVault's generator uses crypto.getRandomValues(), a cryptographically secure source — never Math.random().
  • Prefer length. Four random words are stronger and more memorable than P@ssw0rd!.
  • Never reuse. One breach shouldn't unlock every account — use a password manager.

★ The golden rules

  • Don't roll your own crypto. Use vetted primitives (like the Web Crypto API) and well-reviewed libraries.
  • Encoding ≠ encryption. If there's no key, there's no secrecy.
  • Hash passwords with a slow, salted function — bcrypt, scrypt or Argon2.
  • Authenticate your encryption. Prefer AEAD modes like AES-GCM so tampering is detected.
  • Length > complexity for passwords, and never reuse them.
  • Keep private keys private. Public keys are meant to be shared; private keys never leave your control.
